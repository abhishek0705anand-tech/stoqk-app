Stoqk — System Architecture

Complete technical reference: AI agents, database schema, Inngest jobs, iOS app, data flows, and every architectural decision that shapes the system.

SwiftUI iOS Node.js / Hono Supabase Postgres Inngest Gemini 2.0 Flash NSE India Yahoo Finance APNs Vercel

1. System Overview

Stoqk is an AI-native investing assistant for Indian retail investors. Rather than showing raw price data, it continuously processes NSE market events through specialized AI agents, personalizes the output against each user's portfolio and risk profile, and delivers actionable insights — push notifications, morning briefs, and conversational analysis.

2. Technology Stack

📱 Native iOS (SwiftUI)

Chosen over React Native / Flutter for native Charts framework, Haptics, and UserNotifications. All state managed via @StateObject + Combine. Zero third-party UI libraries.

☁️ Vercel + Hono

Hono is a tiny, fast web framework that runs natively on Vercel's Edge/Serverless runtime. The backend is strictly a data processor — scraping, AI calls, push delivery. No UI concerns.

🗄️ Supabase

Postgres with Row Level Security so the iOS app queries market tables (signals, prices) directly via PostgREST, while user tables are fully locked behind RLS. The backend uses a service-role key that bypasses RLS.

⚙️ Inngest

Vercel functions die at 60 s. Inngest wraps long jobs in step.run() chunks — each chunk is independently retried on failure. No progress is lost. Concurrency limits prevent AI rate-limit storms.

🤖 Google Gemini 2.0 Flash

Selected over GPT-4 / Claude for its 1M-token context window (entire portfolio + news fits), sub-second TTFT, and cost efficiency at scale. All 7 agents use the same model.

📊 NSE India + Yahoo Finance

NSE for real-time deals, insider trades, OHLCV prices, FII/DII flows. Yahoo Finance as a reliable fallback for macro indices and as the primary source for company fundamentals.

5. AI Agents

All agents use Google Gemini 2.0 Flash via lib/anthropic.ts (misnamed — it calls Google's SDK). runAgent<T>() returns structured JSON; streamAgent() streams text for chat.
🔍

Signal Detector Agent

backend/src/agents/signal-detector.ts
JSON output

Role: Receives a raw NSE exchange event (bulk deal, insider trade, block deal) and classifies it — assigning an event type, a significance score, a plain-language summary, and historical context. This is the first gate every market event passes through.

Input
  • Raw NSE event object (deal data, amounts, parties)
  • User profile (experience_level, risk_tolerance, preferred_sectors)
  • Ticker symbol + current price
Output JSON
  • event_type — bulk_deal | insider_buy | insider_sell | pledge_increase | pledge_reduction | promoter_buy | block_deal | qip
  • significance_score — 0–100
  • plain_summary — 2 sentences, calibrated to experience level
  • historical_context — 1 sentence or null
System prompt:
"You are a sharp financial analyst evaluating raw exchange data for Indian markets.
Raw event: {raw_data}
Calibrate language to: {experience_level}
Score rules: base on deal size in crores + % of float + volume deviation vs 30-day avg.
+15 if ticker in user's preferred sectors or top holdings. -10 if aggressive and risk_tolerance=low.
Output strict JSON: { event_type, significance_score (0-100), plain_summary (2 sentences), historical_context (1 sentence if pattern historically preceded a price move, else null) }"

Scoring rubric

  • 80–100: High — deal >₹500Cr, promoter pledge reduction, insider buy of 5%+ float
  • 60–79: Medium — bulk deal ₹50–500Cr, block deal with institutional buyer
  • 0–59: Low — small deal, routine filing, non-material pledge
flowchart LR A[NSE Raw Event] --> B{Signal Detector} B --> C[event_type classification] B --> D[significance_score 0-100] B --> E[plain_summary] B --> F[historical_context] C & D & E & F --> G[INSERT signals table]

Alert Priority Agent

backend/src/agents/alert-priority.ts
JSON output

Role: Given a scored signal and a specific user, decides whether to fire a push notification. Also writes the push copy (≤80 chars) calibrated to the user's experience level.

Input
  • Signal object (event_type, score, summary)
  • User profile (experience_level, risk_tolerance)
  • User's holdings tickers array
  • User's watchlist tickers array
Output JSON
  • should_notify — boolean
  • push_copy — ≤80 character string
  • urgency — high | medium | low
  • reason — explanation string
System prompt:
"User profile: {profile_block}
Signal: {signal_summary} (score: {significance_score})
User holds: {holdings_tickers}. User watches: {watchlist_tickers}.
Rules: Always notify if ticker in top_holdings. Notify if watchlist AND score>50. Notify if score>70.
Urgency: 'high' if score>80, 'medium' if 60–80, 'low' below.
Push copy: ≤80 chars. Beginner: plain English. Advanced/Trader: use price, volume, float %."

Decision matrix

ConditionNotify?Urgency
Ticker is in user's top_holdingsAlwaysDerived from score
Ticker on watchlist AND score > 50YesDerived from score
Score > 70 (any ticker)YesDerived from score
Score ≤ 70 AND not in holdings/watchlistNo—

Push copy examples by experience level

  • Beginner: "A large fund just bought ₹320Cr of your Tata Motors. Usually a good sign."
  • Intermediate: "Block deal: TATAMOTORS ₹320Cr by HDFC MF. Significance score: 74."
  • Trader: "TATAMOTORS block: 1.2Cr sh @ ₹267, HDFC MF. Float: 1.8%. Score: 74."

Daily Brief Agent

backend/src/agents/daily-brief.ts
JSON output

Role: Synthesizes a personalized morning market summary for each user every weekday at 7:30 AM IST. Combines macro data, the user's portfolio performance, and the day's top signals into 6 structured fields.

Input
  • User profile (profile_block)
  • Top 5 signals from past 24 hours
  • Latest macro_indicators row (Nifty, Sensex, FII/DII)
  • Portfolio context JSON (from portfolio-context agent)
Output JSON
  • opening — One striking market fact
  • market_snapshot — Nifty/Sensex move + explanation
  • fii_dii — Net flows in crores
  • top_signal — Highest-score signal summary
  • portfolio_update — Best/worst holding + total PnL (null if no holdings)
  • watch_today — Single most important action sentence
System prompt:
"You are the user's personal wealth manager delivering a morning brief.
Macro: Nifty {nifty_close} ({nifty_change_pct}%), Sensex {sensex_close}. FII: ₹{fii_net_cr}Cr, DII: ₹{dii_net_cr}Cr.
Top signals from past 24h: {signals_block}
Portfolio: {portfolio_context}
Tone: {experience_level == beginner ? 'Reassuring, no jargon, warm' : experience_level == trader ? 'Direct, fast, numbers-first' : 'Factual and clear'}
Output strict JSON with fields: opening, market_snapshot, fii_dii, top_signal, portfolio_update, watch_today."

Stored in daily_briefs

Each user gets one row per date. If they already have a row, it is upserted. The iOS app reads brief_json and renders each field as a separate card section.

Portfolio Context Builder

backend/src/agents/portfolio-context.ts
Pure TypeScript — no AI

Role: Builds a precise mathematical summary of the user's portfolio. Critically — this is not an AI call. The AI was originally asked to compute these numbers and produced hallucinated PnL figures. All math is now done in TypeScript and the result is injected as trusted context into other agents.

Input
  • user_holdings rows (ticker, qty, avg_buy_price)
  • Live prices map (ticker → current_price)
  • Active signals on holding tickers (last 24h)
  • company_fundamentals for sector lookup
Output JSON
  • total_current_value — ₹ sum
  • total_unrealised_pnl — ₹ amount
  • total_unrealised_pnl_pct — %
  • top_gainer — { ticker, pnl_pct }
  • top_loser — { ticker, pnl_pct }
  • sector_exposure — { IT: 45, Banking: 35, … }
  • active_signals_on_holdings — filtered signal array
Why no AI here? During development, Gemini was asked to compute total value from a holdings array. It would round differently, conflate cost basis, or simply produce wrong totals. The portfolio context builder is pure arithmetic — every number is verifiable — then the result is given to the AI as read-only fact.

Sector mapping (hardcoded)

Since live sector data can be slow to fetch, a static map in lib/profile.ts covers all 45 tracked tickers: INFY → IT, HDFCBANK → Banking, SUNPHARMA → Pharma, TATAMOTORS → Auto, etc. Company fundamentals table fills in the rest.

Market Analyst Chat Agent

backend/src/agents/market-analyst-chat.ts
Streaming text output

Role: The conversational core. Takes a user's question and answers in two mandatory layers separated by ---DIG_DEEPER---. Layer 1 is a 1–2 sentence actionable summary. Layer 2 is the full analysis with data, caveats, and news references.

Input
  • User message (question text)
  • profile_block (compiled text)
  • Portfolio context JSON
  • Active signals on holdings (last 7 days)
  • NSE announcements for mentioned tickers
  • NewsAPI headlines for mentioned tickers
Output (streaming text)
[1–2 sentence summary]
---DIG_DEEPER---
[3–5 sentence analysis
with data, caveats, sources]

Backend splits on marker → saves as plain + deeper in chat_messages table.

System prompt:
"You are a sharp market analyst advising a specific Indian investor.
Profile: {profile_block}
Portfolio: {portfolio_context}
Active signals on holdings: {signals_block}
Recent news: {search_results}
Rules:
1. ALWAYS answer in two layers separated by the exact marker '---DIG_DEEPER---'.
2. Layer 1: 1–2 sentence sharp, actionable answer (5-second read).
3. Layer 2: Full analysis with numbers, context, source citations.
4. Never give direct buy/sell advice. Be honest about data gaps.
5. If user asks about a stock you have news for, cite the source."

Chart Pattern Agent

backend/src/agents/chart-pattern.ts
JSON output

Role: Takes a technically-detected pattern (e.g., "Double Top", "Golden Cross") and explains it in plain language appropriate to the user's level. Called by the internal route after the Python pattern scanner detects a pattern via TA-Lib.

Input
  • Ticker symbol
  • Pattern name (from TA-Lib)
  • Last 30 sessions of OHLCV data
  • User profile (experience_level, investment_horizon)
Output JSON
  • plain_explanation — language calibrated to level
  • what_to_watch — specific price level or trigger
  • historical_win_rate — % (researched by AI)
  • avg_move_pct — average expected move
  • horizon_note — relevance to user's investment timeline

Explanation calibration

  • Beginner: Analogy-first, no technical terms. "This pattern looks like the letter M…"
  • Intermediate: Psychology behind the pattern — supply/demand, support/resistance
  • Advanced/Trader: Volume confirmation, RSI divergence, float context, entry/stop levels

6. Inngest Background Jobs

Inngest wraps all long-running work in durable step.run() blocks. If a step fails, only that step retries — no reprocessing from scratch. Each job has a concurrency limit to protect Gemini's rate limits.
Inngest job triggers & schedule (all times IST)
gantt dateFormat HH:mm axisFormat %H:%M section Weekdays signal-fetch (every 15min) :active, 09:15, 3m signal-fetch :active, 09:30, 3m signal-fetch :active, 09:45, 3m price-sync (4pm close) :milestone, 16:00, 0m macro-sync (post-close) :milestone, 18:30, 0m section Morning daily-brief (7:30am) :milestone, 07:30, 0m section Sunday fundamental-sync (10am) :milestone, 10:00, 0m

signal-fetch

Cron: */15 * * * 1-5 Every 15 min weekdays concurrency: 1
  1. Fetch bulk deals, insider trades, block deals from NSE (3 parallel API calls)
  2. De-duplicate against existing signals in DB (filter by ticker+date+type)
  3. For top 20 new events: 5s delay between each (rate-limit pacing)
  4. Call signal-detector agent → get event_type, score, summary
  5. INSERT row into signals table
  6. SELECT users who hold or watch this ticker (from user_holdings + user_watchlist)
  7. For each relevant user: call alert-priority agent
  8. If should_notify=true: INSERT user_notifications row
  9. Fetch user's APNs tokens from user_devices
  10. Call lib/apns.ts sendPushToMany()

daily-brief

Cron: 0 2 * * 1-5 7:30 AM IST weekdays up to 500 users/run
  1. Fetch latest macro_indicators row from DB
  2. Fetch top 5 signals from past 24 hours (ORDER BY significance_score DESC)
  3. SELECT all users WHERE onboarding_completed = true (max 500)
  4. For each user: fetch user_holdings + live prices via NSE API
  5. If holdings exist: call buildPortfolioContext() (TypeScript math, no AI)
  6. Call daily-brief agent with macro + signals + portfolio context + profile
  7. UPSERT row in daily_briefs (conflict on user_id + date)

chat-process

Event: app/chat.message Triggered by POST /chat concurrency: 5
  1. Receive event payload: { job_id, user_id, message }
  2. Fetch user profile + holdings from Supabase
  3. Fetch live prices for all user holdings via NSE API
  4. Fetch active signals on holdings (last 7 days, limit 10)
  5. Call buildPortfolioContext()
  6. Extract mentioned tickers from message (simple regex, max 3)
  7. For each ticker: fetch NSE announcements + NewsAPI headlines (last 7 days)
  8. Call market-analyst-chat agent (streaming)
  9. Collect all stream chunks into fullText
  10. Split on ---DIG_DEEPER--- → plain + deeper strings
  11. UPDATE chat_messages: plain, deeper, status="done", completed_at=now()
  12. On 429 rate limit error: wait 2s, retry once

price-sync

Cron: 0 16 * * 1-5 4:00 PM IST weekdays (post-close)

Syncs 180 days of OHLCV data for all 45 tracked NSE tickers. Uses NSE's GetQuoteApi endpoint with session cookie spoofing. UPSERT conflict on (ticker, date) — safe to re-run.

Tracked tickers: RELIANCE, TCS, HDFCBANK, INFY, ICICIBANK, HINDUNILVR, BHARTIARTL, ITC, SBIN, BAJFINANCE, KOTAKBANK, AXISBANK, LT, ASIANPAINT, MARUTI, WIPRO, SUNPHARMA, ULTRACEMCO, TITAN, NESTLEIND, and 25 more NSE 500 constituents.

macro-sync

Cron: 30 12 * * 1-5 6:00 PM IST weekdays

Fetches Nifty 50 (^NSEI) and Sensex (^BSESN) closing prices from Yahoo Finance. Fetches FII/DII net flows from NSE's /api/fiidiiTradeReact endpoint. UPSERT on today's date — only one row per day.

fundamental-sync

Cron: 0 10 * * 0 10:00 AM IST every Sunday concurrency: 1

Fetches PE ratio, ROE, debt/equity, revenue growth, promoter holding, market cap, and sector from Yahoo Finance for all tickers in company_fundamentals. Processes in batches of 20 with 300ms delays to avoid Yahoo rate limits.

Signal — full lifecycle

  1. Origin: NSE API returns bulk deal / insider trade JSON every 15 minutes
  2. Detection: signal-detector agent scores it (0–100), classifies event_type, writes plain_summary
  3. Storage: INSERT into signals table with raw_data, score, summary, detected_at
  4. Alert: alert-priority agent checks each user's holdings/watchlist → push notification if relevant
  5. Brief: Next morning, top 5 signals of the past 24h are included in every user's daily brief
  6. Chat: Active signals on user's holdings are included in every chat session as context
  7. Radar feed: iOS app queries GET /signals to display paginated signal feed
  8. Performance tracking: Backfill job (manual) checks perf_1d/7d/30d after respective periods

profile_block — full lifecycle

  1. Created: profile-builder agent converts onboarding answers → compact English paragraph
  2. Stored: Saved as profile_block TEXT column in user_profiles
  3. Updated: Rebuilt whenever user updates their onboarding answers or portfolio_size_bucket changes
  4. Injected: Every AI agent call fetches getProfileWithBlock(userId) and prepends profile_block to the system prompt
  5. Result: The same signal scored a 45 for a high-risk trader might score 78 for a low-risk beginner holding the stock — because the profile_block changes the AI's evaluation context

12. Key Architectural Decisions

A. TypeScript math, not AI math

The AI was originally asked to compute portfolio total value, PnL, and sector concentration from raw holdings arrays. It frequently hallucinated wrong totals, misapplied cost basis, or truncated the JSON mid-response.

Decision: buildPortfolioContext() is pure TypeScript arithmetic — deterministic, testable, exact. These verified numbers are then injected as read-only facts into agent prompts. The AI only interprets, never computes.

B. Inngest steps instead of monolithic functions

Processing 40 signals × 100 users = ~4,000 AI calls in one run. A single Vercel function times out at 60 seconds and leaves no recovery point.

Decision: Each signal evaluation is wrapped in its own step.run(). If Gemini rate-limits at signal #23, Inngest pauses, waits, and resumes at signal #23. The first 22 are not re-processed. The 5-second delay between signals also spreads load naturally.

C. Two-layer chat response

Financial LLM answers tend to be walls of text. Most users want a fast answer, not a full analysis every time.

Decision: The chat agent is forced to structure every response around the exact string ---DIG_DEEPER---. The backend splits on this marker. The iOS app receives plain (1–2 sentences) and deeper (full analysis). Layer 1 is shown immediately; Layer 2 is behind a "Dig Deeper" disclosure group. The iOS app never needs to know how the split happens.

D. NSE session cookie spoofing

NSE India's APIs return 403/404 to direct server requests. They require a browser-like session.

Decision: lib/nse.ts implements an ensureNSESession() function that fetches the NSE homepage first to obtain a valid session cookie, then reuses it for up to 5 minutes. The session is cached in memory between requests in the same Vercel function invocation.

E. profile_block as a compiled string

Passing structured JSON for the user profile into every prompt led to the AI occasionally ignoring nested fields or treating them inconsistently.

Decision: The profile is compiled once into a human-readable paragraph (profile_block) and stored in the database. This pre-rendered text is injected directly into system prompts. The AI parses natural language more reliably than it interprets arbitrary JSON schemas.

F. Polling over WebSockets for chat

Real-time WebSocket connections complicate serverless deployments — Vercel functions are stateless and short-lived.

Decision: The chat endpoint immediately returns a job_id. The iOS app polls GET /chat/result/:id every 1.5 seconds for up to 60 seconds. This is simple, reliable, and works within Vercel's execution model. The polling loop stops the moment status is "done" or "error".

13. External Services & Environment

Google Gemini 2.0 Flash

All 7 AI agents. GEMINI_API_KEY. SDK: @google/generative-ai. Both FAST_MODEL and SMART_MODEL point to gemini-2.0-flash. Used for: runAgent (JSON), streamAgent (streaming text).

Supabase

SUPABASE_URL, SUPABASE_SERVICE_ROLE_KEY, SUPABASE_ANON_KEY. Service role for all backend writes. Anon key for auth validation in middleware.

Inngest

INNGEST_SIGNING_KEY, INNGEST_EVENT_KEY. Webhook at /api/inngest. Manages cron schedule, event triggers, concurrency, and retries.

Apple APNs

APNS_KEY_ID, APNS_TEAM_ID, APNS_PRIVATE_KEY (.p8 contents), APNS_BUNDLE_ID, APNS_ENV. SDK: apn npm package. Dynamic import to minimize cold-start cost.

NSE India APIs

No key required. Requires browser-session spoofing. Endpoints: bulk deals, insider trades (PIT), block deals, OHLCV (GetQuoteApi), live quote, FII/DII flows, corporate announcements.

Yahoo Finance

SDK: yahoo-finance2. Symbols: ^NSEI (Nifty 50), ^BSESN (Sensex). Company fundamentals: quoteSummary() with financialData + defaultKeyStatistics modules.

NewsAPI

NEWS_API_KEY. Used in chat agent to fetch recent headlines for mentioned tickers. Free tier: 100 requests/day. Endpoints: /v2/everything with company name + ticker query.

Upstox API

UPSTOX_ACCESS_TOKEN (optional). Used for batch live price quotes. Falls back to NSE /api/quote-equity if token not available.

Internal API Key

The INTERNAL_API_KEY is a shared secret between the backend and the Python pattern scanner service. Internal routes check for x-internal-key header. This prevents the pattern analysis endpoint from being called by external actors.
